# DayLog 需求文档（v1.0 定稿）

> 一款本地优先的 Windows 日程记录工具：随手用自然语言记录，一键生成日报、周报、季度总结、年度总结，支持导入自定义 Markdown / Word 模板并导出为 Word。

- 文档状态：已敲定，作为 v1.0 开发依据
- 日期：2026-06-12

---

## 1. 产品定位

### 1.1 一句话描述
给个人（尤其是需要定期汇报的职场人）用的"流水账 → 正式报告"转换器：平时零负担地记，汇报时零负担地出稿。

### 1.2 目标用户
- 需要写日报/周报的工程师、运营、项目经理
- 需要做季度/年度总结的所有职场人
- 想留工作痕迹但讨厌格式约束的人

### 1.3 核心原则（产品取舍的依据）
1. **记录零门槛**：打开就是输入框，写完即存，永远不弹"请选择分类"之类的表单。
2. **本地优先**：所有数据存在本机，不注册、不登录、离线全功能可用（AI 总结除外）。
3. **界面极简**：全应用只有 3 个页面 + 1 个设置弹窗，无多余功能。
4. **架构极简**：单进程桌面应用 + 单文件数据库，无服务端、无后台服务。

### 1.4 明确不做（v1.0 范围外）
- 多端同步 / 移动端 / Web 版
- 多用户、团队协作、分享链接
- 提醒、闹钟、待办管理（这是记录工具，不是任务管理器）
- 富文本编辑器（统一用 Markdown 纯文本）
- 图片/附件（v1.1 再考虑）

## 2. 技术选型（已定）

| 项 | 选型 | 理由 |
|---|---|---|
| 桌面框架 | **Tauri 2** | 产物是单个原生 exe，安装包 < 15MB；Win10/11 自带 WebView2，无需捆绑浏览器内核（对比 Electron 100MB+） |
| 前端 | **React 18 + Vite + TypeScript** | 三个页面的体量，够用且简单；不引入 UI 框架全家桶，只用少量手写组件 + CSS；状态轻量，用 React 自带 hooks 即可，不需要 Redux |
| 数据存储 | **SQLite 单文件** | 路径 `%APPDATA%\daylog\daylog.db`；用户拷走一个文件就完成备份/迁移 |
| 后端逻辑 | Tauri Rust 侧（薄层） | 只做：SQLite 读写、文件导入导出、调用 LLM API；业务逻辑尽量放前端 |
| AI 能力（可选） | **单一 OpenAI 兼容端点** | 本地模型（Ollama / LM Studio / vLLM）或云端服务皆可，填 Base URL + Key（见 3.7 节）；**不配置时所有核心功能照常可用** |
| 打包分发 | Tauri bundler | **硬性要求：产物为单文件 exe**，详见 2.1 节 |

**架构图：**

```
┌─────────────────────────────────┐
│  DayLog.exe (Tauri, 单进程)      │
│                                 │
│  React 前端 (WebView2)           │
│   ├─ 记录页  ├─ 回顾页  ├─ 报告页 │
│        │ invoke                 │
│  Rust 薄层                       │
│   ├─ SQLite 读写                 │
│   ├─ 模板/导出文件 IO             │
│   └─ AI 调用（OpenAI 兼容端点，可选）   │
└──────────┬──────────────────────┘
           ▼
  %APPDATA%\daylog\
   ├─ daylog.db        (数据)
   ├─ templates\*.md   (模板)
   └─ settings.json    (配置)
```

### 2.1 单文件 exe 分发（硬性要求）

Tauri 构建时把全部前端资源（HTML/JS/CSS）和 SQLite（rusqlite 静态链接）编译进二进制，`cargo tauri build` 直接产出一个**无任何外部依赖文件**的 `DayLog.exe`（约 8~12MB）。

**部署约束（已确认）：目标机器可能完全离线**，因此分发不得依赖任何联网下载步骤。发布两个形态：

1. **绿色版**：`DayLog-x.y.z.exe` 单文件（约 8~12MB），双击即用，无需安装。
   - 适用前提：机器上已有 WebView2 运行时（Windows 11 全部自带；打过近年更新的 Win10 也自带）。
   - 启动时若检测不到 WebView2，弹窗提示「请改用离线安装版」（不做联网引导，离线环境引导无意义）。
   - 便携数据规则：启动时若 exe 同目录存在 `data\` 文件夹，则数据库/模板/配置全部存在 `data\` 里（U 盘便携模式）；否则使用 `%APPDATA%\daylog\`。
2. **离线安装版（离线环境主推）**：`DayLog-Setup-x.y.z-offline.exe`，单个安装 exe（约 140MB）。
   - Tauri 配置 `bundle.windows.webviewInstallMode = "offlineInstaller"`，把微软官方 WebView2 离线安装器完整内嵌进 NSIS 安装包，安装全程**零网络请求**。
   - 已有 WebView2 的机器上会自动跳过该步骤。
   - 同时提供开始菜单快捷方式 + 卸载入口。

**结论**：两个产物都是"一个 exe"；断网的全新 Win10 机器用形态 2 一次装齐，Win11 / 已有 WebView2 的机器直接用形态 1 的 8~12MB 单文件。

**离线环境下的功能边界**：核心功能（记录、回顾、模板、报告"直接整理"、全部导出）100% 离线可用；AI 功能（AI 总结、多日拆分）在离线环境下可通过**本地模型**（3.7 节）使用；仅当 Base URL 指向云端服务时才需要联网。未配置 AI 服务时，AI 相关入口置灰并注明原因。

## 3. 功能需求

### 3.1 记录（主页面）

**输入方式**
- 页面主体是一个多行文本框，placeholder：「这会儿做了什么？随便写…」
- `Ctrl+Enter` 保存为一条记录；保存后清空输入框，焦点保留。
- 无任何格式要求、无必填项、无长度下限；上限 10,000 字符。
- 每条记录自动附加创建时间戳（精确到分钟）。

**自然语言的轻量解析（本地正则，无需联网）**
保存时自动提取以下标记，**提取失败不报错、不打扰**，原文永远完整保留：
- `#标签` → 存入 tags（如 `#会议 #踩坑`），用于过滤和报告分组
- `@项目名` → 存入 project（如 `@支付系统`）
- 不解析时间表达式（"下午3点开会"原样保留为正文，时间一律以记录时刻为准——这是刻意简化，避免解析错误带来的纠错负担）

**名称识别规则（已明确）**
- `#` 或 `@` 之后、**到首个空白或标点为止**的一段连续字符即为标签/项目名；该段之后的内容（含触发的标点）不计入名称、仍是正文。
- 作为分隔的标点包含中英文常见符号：空白、`#`、`@`、冒号（`:` `：`）、逗号（`,` `，`）、句号（`.` `。`）、顿号、分号、引号、括号、问号、感叹号、斜杠等。
- 名称内允许字母、数字、汉字、下划线、连字符等非标点字符。
- 例：`@AI：编写需求矩阵` → 项目名是 `AI`（`：编写需求矩阵` 为正文）；`#后端联调 完成` → 标签是 `后端联调`。

**标签与项目的管理策略（已确认：隐式管理 + 自动补全）**
- **隐式管理，零维护**：无需预先创建标签/项目；候选列表由全库去重动态生成，用了就存在、无引用即消失。
- **不提供重命名/合并**：标签嵌在记录原文中，重命名意味着批量改写历史原文，与"原文永不改写"原则冲突，v1.0 刻意不做。
- **输入自动补全（从源头保证一致性）**：在输入框键入 `#` 或 `@` 时，弹出已有标签/项目的前缀匹配下拉（按使用频次排序），`↑↓` 选择、回车/点击补全、`Esc` 关闭；记录页输入框、当日详情面板的补记框、回顾页搜索框均生效。这是避免「#联调 / #连调」之类拼写分裂的主要手段。

**当日列表**
- 输入框下方实时显示"今天"的记录列表（倒序），每条可悬停出现 编辑 / 删除 按钮。
- 编辑为就地编辑（textarea 替换原文本），`Ctrl+Enter` 保存、`Esc` 取消。
- 删除需二次确认（行内确认按钮，不弹模态框）。

**补记**
- 输入框右下角有一个不起眼的日期按钮（默认"今天"），点开日历可改为过去某天，用于补记。补记条目的时间戳为所选日期 + 保存时刻的时间。

**多日批量录入（AI 拆分，需配置 AI 提供方）**

用户可以一次性输入多天的工作内容（如周五补一周的账），由 AI 按日期拆分成多条记录，**经用户确认后**才入库：

- **触发（已确认：手动按钮，不自动检测）**：输入框工具条上的「AI 拆分多日…」按钮；未配置任何 AI 提供方（3.7 节）时置灰，悬停提示去设置。
- **流程**：输入框原文 → 发送 AI（系统提示要求输出 JSON 数组 `[{date, content}]`）→ 弹出拆分确认界面 → 用户调整确认 → 按日期分别入库。
- **对 AI 的硬约束（写入系统提示）**：
  - `content` 必须是原文的连续片段，**不改写、不润色、不增删信息**；仅允许剥离行首的纯日期引导词（如"周一："）。
  - 日期解析以今天为锚点，取**不晚于今天的最近日期**（周五说"周一"= 本周一；"昨天"= 今天-1）。
  - 无法确定日期的片段必须标记为未识别，禁止猜测。
- **拆分确认界面**（模态）：左侧为原文只读对照；右侧为按日期分组的条目列表，每条可改日期（日期选择器）、可编辑文本、可删除，可手动新增条目；**未识别日期的条目仅以黄色高亮标识（不显示"未识别"等文字标注），默认归到今天**，含义由底部图例说明；用户修改其日期后高亮消除；条目文本框高度**自适应内容**（随编辑自动伸缩，不提供手动拖拽调整）；底部「确认导入 N 条」/「取消」。
- **入库规则**：与补记一致——`entry_date` = 拆分出的日期，`created_at` = 导入时刻；不伪造当时的时间。入库后正常走 #标签/@项目 提取。
- **失败处理**：AI 返回非法 JSON 或超时 → 顶部横幅报错；输入框原文原样保留，不丢内容。

### 3.2 回顾（日历页面）

**展示方式：月历网格（已确认，替代时间线方案）**

- 主体为标准月历：7 列（周一为每周第一天，与周报口径一致）× 4~6 行，铺满页面。
- 每个日期格子内直接展示当天记录的**原始自然语言文本**：
  - **不做任何格式化处理**——不渲染 Markdown、不高亮 #标签/@项目、不截词改写，逐字显示用户当时输入的内容（v1.0 刻意如此，后续版本再考虑格式化展示）。
  - 每条记录占一行，前缀显示时间（`14:32 和支付系统联调完成…`），单行放不下时尾部省略号截断。
  - 格子高度有限时最多显示 3 条 + `+N 条` 角标。
- 点击任意日期格子，右侧滑出当日详情面板：完整展示该天全部记录原文（同样不格式化），可编辑/删除（交互与记录页一致），可在该面板内直接补记当天条目。
- 无记录的日期格子留空（不显示"无记录"之类的占位文案）；非当月的补位日期灰显。
- 今天的格子有强调色边框。

**分页与导航（按年、月）**

- 顶栏：`◀ 上月`、`2026 年 ▾`（年份下拉，范围 = 数据库中有记录的年份）、`6 月 ▾`（月份下拉）、`下月 ▶`、`回到今天` 按钮。
- 切换年/月即整页翻页，每次只加载并渲染当月数据（单月查询，天然分页，无性能压力）。
- 快捷键：`←` / `→` 翻月，`PageUp` / `PageDown` 翻年。

**搜索与过滤（保留，形态简化）**

- 顶栏右侧为单个搜索框：自由文本按内容匹配，键入 `#` 或 `@` 弹出标签/项目补全（与记录页同一套补全），可混写（如 `联调 @DayLog`）叠加过滤。不再单独罗列全部标签/项目的下拉。
- 输入搜索词或选中过滤条件时，日历暂时切换为扁平的匹配结果列表（按日期倒序、原文显示，点击某条可跳回其所在月份的日历）；清空条件即恢复日历视图。

### 3.3 报告生成（报告页面）

**报告类型与默认时间范围**

| 类型 | 默认范围 | 范围可调 |
|---|---|---|
| 日报 | 今天 | 任选某一天 |
| 周报 | 本周一 ~ 今天 | 任选某一周（周一为一周起点） |
| 季度总结 | 本季度（1-3/4-6/7-9/10-12月） | 任选某季度 |
| 年度总结 | 今年 1/1 ~ 今天 | 任选某一年 |

**生成流程（3 步，单页面完成）**
1. 选类型 → 自动带出默认时间范围（可改）。**范围调整一律通过选择器完成，不提供手动文本输入**（避免格式错误），控件随类型切换：
   - 日报：日期选择器（datepicker）选某一天
   - 周报：日期选择器点选该周内任意一天，自动换算为 周一 ~ 周日（当周则截至今天），换算结果实时回显
   - 季度总结：年份下拉 + 季度下拉（Q1~Q4）
   - 年度总结：年份下拉（年份范围 = 数据库中有记录的年份）
2. 选模板（下拉，默认选中该类型的内置模板）
3. 选生成方式：
   - **直接整理**（默认，离线）：把范围内的记录按模板占位符填充，原文呈现，按天/按标签分组。
   - **AI 总结**（需配置 AI 提供方，见 3.7 节）：把范围内记录 + 模板发给所配置的模型，生成提炼后的报告。未配置时此选项置灰并提示去设置。

**生成结果**
- 右侧实时预览（Markdown 渲染）。
- 结果可直接在预览区编辑（编辑的是 Markdown 源文本，左源右预览）。
- 操作按钮：复制到剪贴板（Markdown / 纯文本两种）、导出文件。

**AI 总结的约定**
- 请求体 = 系统提示（固定）+ 模板内容 + 时间范围内全部记录原文，发往所配置的 AI 提供方（3.7 节）。
- 系统提示要点：以模板结构为骨架；只基于记录内容，不编造；中文输出；保留 #标签 涉及的关键信息归类。
- 记录超长时截断并在结果顶部注明（当前实现按 30000 字符上限截断）。
- 隐私：是否出网取决于用户填写的 Base URL（本地地址不出网、云端地址才联网），由用户自行掌握，不再单独弹云端确认。
- 失败处理：网络/Key/本地服务未启动等错误用顶部横幅提示，自动回退按钮「改用直接整理」。

### 3.4 模板系统

支持两类模板：Markdown 模板（占位符填充）与 Word 模板（样式 + 结构填充）。

#### 3.4.1 Markdown 模板

**模板格式**：UTF-8 的 Markdown 文件，内嵌 `{{占位符}}`。

**占位符清单（v1.0 全集，敲定）**

| 占位符 | 含义 |
|---|---|
| `{{date}}` | 报告生成日期（YYYY-MM-DD） |
| `{{range}}` | 时间范围（如 2026-06-08 ~ 2026-06-12） |
| `{{entries}}` | 全部记录，按天分组、天内按时间排序的 Markdown 列表 |
| `{{entries_by_tag}}` | 按 #标签 分组的记录（无标签的归入"其他"） |
| `{{entries_by_project}}` | 按 @项目 分组的记录（无项目的归入"其他"） |
| `{{summary}}` | AI 总结正文；「直接整理」模式下该占位符替换为 `{{entries}}` 的内容 |
| `{{stats}}` | 统计行：共 N 条记录，覆盖 M 天，标签 Top5 |

**模板文件头（YAML frontmatter，可选）**
```markdown
---
name: 周报-汇报版
type: weekly        # daily | weekly | quarterly | yearly | any
---
# {{range}} 周报

## 本周工作
{{summary}}

## 数据
{{stats}}
```

**内置模板**：日报、周报、季度总结、年度总结各 1 个，首次启动写入模板目录，用户可改可删（删了下次启动不会恢复，设置里有"恢复内置模板"按钮）。

#### 3.4.2 Word 模板

**模板格式**：`.docx` 文件，提供样式与固定结构，不含占位符。模板可包含标题、各级章节标题、表格、页眉页脚等排版元素，章节正文里可写"写作说明"指引 AI 撰写。

**解析与填充规则**
- 按段落样式（heading 1 / heading 2 等）切分章节：首个标题段为报告标题，其后每个标题段起一节。
- 含表格的章节为"表格章节"，纯段落的章节为"正文章节"。
- 填充时保留模板全部排版：标题套用时间范围（年份 / 月份 / 第几周）；表格保留表头与标签行，仅替换空数据行并按需追加行；正文章节用生成内容替换说明段落。
- 生成方式：表格行与正文均可由「AI 总结」按章节结构与写作说明生成；「直接整理」为离线兜底，将记录粗略填入表格与正文。
- 屏幕预览以等价 Markdown 呈现，实际产物为 `.docx` 字节。

#### 3.4.3 导入与管理
- 报告页的模板下拉里有「导入模板…」项：文件选择器选 `.md` 或 `.docx` 文件，复制进 `%APPDATA%\daylog\templates\`。
- 设置弹窗里有「打开模板文件夹」按钮，高级用户直接用文件管理器增删改（应用每次进入报告页重新扫描该目录）。
- Markdown 模板解析失败（如 frontmatter 格式错）则在下拉中标灰并注明"格式错误"；Word 模板在下拉中标注「（Word）」。

### 3.5 导出

| 形式 | 说明 |
|---|---|
| 复制到剪贴板 | Markdown 源文 / 去格式纯文本 |
| 导出 .md 文件 | 默认文件名为英文且体现周期序号：日报 `Daily_2026-06-12.md`、周报 `Weekly_2026-W24.md`（ISO 周数）、季度 `Quarterly_2026-Q2.md`、年度 `Yearly_2026.md` |
| 导出 .html | 内联样式的独立 HTML，可直接发邮件/转 PDF |
| 导出 .docx | 仅 Word 模板可用：把记录回填进模板，导出保留原样式与结构的 Word 文档，文件名同上（如 `Weekly_2026-W24.docx`） |
| 导出 PDF | 调用 WebView 打印对话框（系统"另存为 PDF"），不自研 PDF 引擎 |
| 全量数据导出 | 设置弹窗内：导出全部记录为一个 JSON（备份用）；对应有 JSON 导入（合并，按条目 UUID 去重） |

### 3.6 设置（弹窗，非独立页面）

- AI 服务（详见 3.7 节）：单一 OpenAI 兼容端点——Base URL（默认 `http://localhost:11434/v1`）+ API Key（可选，密码框，存 settings.json，仅本机）+ 模型名；外加「测试连接」按钮（仅凭 Base URL + Key 即可测试，结果就地显示在按钮右侧，成功后模型名变为端点返回模型的下拉框）。不再有"Claude / 本地"提供方切换。
- 模板：打开模板文件夹、恢复内置模板
- 数据：打开数据文件夹、导出全部 JSON、导入 JSON
- 外观：浅色 / 深色 / 跟随系统（默认跟随系统）
- 关于：版本号、检查更新（仅打开 GitHub Releases 页面，不做自动更新）

### 3.7 AI 服务（统一单端点）

AI 总结（3.3 节）与多日拆分（3.1 节）共用同一处配置——**单一 OpenAI 兼容端点**，不再区分"Claude / 本地"两种提供方：

| 配置项 | 说明 |
|---|---|
| **Base URL** | OpenAI 兼容接口地址，默认 `http://localhost:11434/v1`；对接 `/v1/chat/completions`（聊天）与 `/v1/models`（测试连接列模型） |
| **API Key** | 可选；本地模型通常留空，云端服务填对应 Key（密码框，存 settings.json，仅本机） |
| **模型名** | 如 `qwen3:14b`（本地）或云端模型名；**测试连接成功后此项变为下拉框**，可直接从端点返回的模型列表中选择（自动默认选中其一） |

- **本地模型**（Ollama / LM Studio / vLLM）：填本机地址，数据不出本机，**完全离线可用**。
- **云端服务**：任何提供 OpenAI 兼容端点的服务（含 Anthropic 的 OpenAI 兼容地址）填对应 Base URL + Key 即可，需联网。
- Rust 侧只实现**一个** OpenAI 协议薄客户端；系统提示与上层逻辑完全共用。一套代码通吃本地与云端。
- 「测试连接」按钮：**仅需填写 Base URL（API Key 可选）即可测试，无需先填模型名**——调用端点的 `/v1/models` 探活并取回可用模型列表。
  - 连接结果**就地显示在按钮右侧、靠右对齐**（成功为绿色并标注模型数量，失败为红色并附错误信息；不再用一次性浮层提示）。
  - 成功后「模型名」一项由文本输入框变为**下拉框**，列出端点返回的模型并默认选中其一；改动 Base URL / API Key 会清空上次结果、模型名回退为文本输入框。
- **是否启用判定**：Base URL 与模型名都填写即视为已配置。未配置时所有 AI 入口（AI 总结、AI 拆分按钮）置灰并提示去设置，核心功能不受任何影响。

## 5. 数据模型（SQLite，已定）

```sql
CREATE TABLE entries (
  id         TEXT PRIMARY KEY,          -- UUID v4
  content    TEXT NOT NULL,             -- 原文，永不改写
  tags       TEXT NOT NULL DEFAULT '[]',-- JSON 数组，如 ["会议","踩坑"]
  project    TEXT,                      -- @项目，可空
  entry_date TEXT NOT NULL,             -- 归属日 YYYY-MM-DD（支持补记）
  created_at TEXT NOT NULL,             -- ISO8601 本地时间
  updated_at TEXT NOT NULL
);
CREATE INDEX idx_entries_date ON entries(entry_date);

CREATE TABLE reports (                  -- 生成历史（最多留最近 50 份）
  id         TEXT PRIMARY KEY,
  type       TEXT NOT NULL,             -- daily|weekly|quarterly|yearly
  range_start TEXT NOT NULL,
  range_end  TEXT NOT NULL,
  template   TEXT NOT NULL,             -- 模板名快照
  content    TEXT NOT NULL,             -- 最终 Markdown（含用户手改）
  created_at TEXT NOT NULL
);
```

- 删除记录为物理删除（产品上已有二次确认；备份靠 JSON 导出）。
- 时区一律取系统本地时区；`entry_date` 决定记录归属哪天。

## 6. 非功能需求

| 项 | 指标 |
|---|---|
| 启动时间 | 冷启动 ≤ 2s（普通 SSD 机器）；启动期间显示闪屏，不留白屏（见 4 节「启动闪屏」） |
| 体积 | 绿色版单文件 ≤ 15MB；离线安装版 ≤ 160MB（内嵌 WebView2 离线运行时所致） |
| 内存占用 | 空闲时 ≤ 150MB |
| 数据规模 | 10 万条记录下搜索 ≤ 500ms |
| 系统要求 | Windows 10 1803+ / Windows 11；WebView2 缺失时由离线安装版内嵌的官方离线安装器补齐，全程无需联网 |
| 离线可用性 | 安装与全部核心功能在完全断网环境下可用；AI 功能填本地模型 Base URL 后亦可离线使用，仅 Base URL 指向云端时需联网 |
| 隐私 | 是否出网取决于用户填写的 Base URL（本地走 localhost、不出网；云端地址才联网）；其余零网络请求；无遥测、无统计上报 |
| 数据安全 | 写库用事务；启动时若检测到 db 损坏，提示并自动备份原文件为 `.bak` |

## 8. 主要风险与对策

1. **用户记得太随意，AI 总结质量差** → 系统提示中要求"信息不足处直说，不编造"；模板引导用户结构。
2. **WebView2 缺失（老 Win10）+ 目标机器离线** → 离线安装版内嵌微软官方 WebView2 离线安装器（`offlineInstaller` 模式），安装全程零联网；代价是该安装包约 140MB，可接受。
3. **正则解析误伤**（如代码片段里的 `#`）→ 解析只影响标签索引，原文永远不被改写，误伤代价≈0。
4. **API Key 明文存储** → v1.0 接受（本机单用户场景），设置页注明；v1.1 评估 Windows DPAPI 加密。
5. **本地小模型质量不稳（拆分错日期 / 总结跑偏）** → 拆分必经人工确认界面兜底，未识别日期高亮而非瞎猜（系统提示明令禁止猜测）；总结结果本就可编辑；设置中可随时更换 Base URL / 模型对比效果。
